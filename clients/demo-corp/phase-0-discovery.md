## Problem Statement

Demo Corp's engineering team (50 engineers, 8 teams) deploys 50+ times per week across 12 microservices. Deployment tracking is currently manual: engineers update a shared Google Sheet after deploying, post in #deploys on Slack, and move Jira tickets by hand. This process breaks constantly:

- 30% of deploys are never logged in the spreadsheet
- Jira tickets sit in "In Review" for days after code is already in production
- "Did that deploy?" is the most common question in #standup (asked 3-4x daily)
- Incident response requires checking 4 systems manually to determine what changed
- Sprint velocity metrics are unreliable because ticket transitions lag behind actual work

The CTO (Sarah Chen) estimates the team loses 15-20 engineer-hours per week to deployment coordination overhead.

## Proposed Solution

Automated deployment pipeline tracking that connects Jira, GitHub, AWS CodePipeline, and Slack into a single event stream:

1. **Capture** — Webhooks from Jira (ticket transitions), GitHub (PRs, merges), and CodePipeline (deploy events) flow into a webhook-router service
2. **Correlate** — Events are linked by Jira ticket key (extracted from branch names and PR titles via `DEMO-\d+` pattern)
3. **Track** — Deploy-service maintains pipeline state: ticket created → PR opened → CI passed → deployed to staging → deployed to prod
4. **Notify** — Notification-service posts updates to Slack (#deploys channel) and updates Jira tickets automatically
5. **Report** — Dashboard shows deployment frequency, lead time, failure rate, and recovery time (DORA metrics)

Target: zero manual deployment tracking. Engineers merge PRs and everything else happens automatically.

## Scope Boundaries

### In Scope

- Webhook-router service: ingest, normalize, and route events from Jira, GitHub, CodePipeline
- Deploy-service: pipeline state tracking, event correlation by Jira ticket key
- Notification-service: Slack notifications with thread management, Jira auto-transitions
- DORA metrics dashboard (deploy frequency, lead time, failure rate, MTTR)
- Teams meeting transcript ingestion for sprint retro context
- LaunchDarkly integration for feature flag tracking per deploy
- Runbook: how to add a new service to the pipeline

### Out of Scope

- Changing Demo Corp's CI/CD pipeline itself (CodePipeline stays as-is)
- Migrating off Jira or Slack
- Building a custom deployment tool (we track, not orchestrate)
- Mobile notifications (Slack mobile covers this)
- Historical data migration (start tracking from go-live date)
- Confluence migration to Notion (separate initiative)

## Risk Matrix

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Jira webhook delivery failures | Medium | High | Webhook-router stores last-seen event ID. Periodic reconciliation job polls Jira for missed events every 5 minutes. |
| Slack rate limiting during burst deploys | High | Medium | Queue messages with 1.1s spacing. Batch notifications for simultaneous deploys. |
| CodePipeline event delay (30-90s) | High | Low | Acceptable delay for tracking. Dashboard shows "last updated" timestamp. Users expect near-real-time, not real-time. |
| Branch naming convention not followed | Medium | Medium | Correlation falls back to PR title regex. If neither matches, event logged as "unlinked" for manual review. |
| Okta SSO outage blocks dashboard access | Low | Medium | Dashboard has read-only cache mode. Cached data serves for up to 1 hour without auth. |
| Redis failover causes auth blip | Medium | Medium | Circuit breaker on Redis reads. Fallback: skip cache, validate token directly with Okta (slower but functional). |
| GitHub webhook secret rotation | Low | High | Documented runbook. Secret stored in Secrets Manager. Rotation script updates both sides atomically. |

## Success Criteria

1. Zero manual deployment logging within 2 weeks of go-live
2. "Did that deploy?" questions eliminated from #standup (measurable: search Slack history)
3. Jira ticket transitions happen within 2 minutes of the corresponding deploy event
4. DORA metrics dashboard live and accurate for all 12 services
5. Incident response: "what changed in the last hour?" answerable in under 10 seconds from dashboard
6. Sprint velocity accuracy improves by 40% (measured by comparing Jira close dates to actual deploy dates)
7. Team satisfaction survey: >80% of engineers say deployment tracking is "no longer a problem"
